空间（Memory）**的计算逻辑不太一样。
简单来说：Merge Sort 是因为需要一个“额外的空房间”来放数据；而 QuickSort 只是需要几个“记事本”来记住递归到哪了。
1. 为什么 Mergesort 是 O(N)？
归并排序最核心的操作是 merge(left, right)。
    必须要有辅助空间：当你合并两个有序序列时，你不能在原来的数组里直接原地交换。比如合并 [5, 8] 和 [1, 9]，当你发现 1 应该放在最前面时，如果你直接把 1 挪到 5 的位置，5 就没地方放了，会被覆盖掉。
    分配一个临时数组：为了不丢失数据，你必须开辟一个和原数组一样大的辅助数组（Auxiliary Array）。先把合并好的结果一个个放进这个临时数组，等合并完了，再拷贝回原数组。
    空间大小：这个辅助数组的大小就是 N。即使在递归过程中会产生很多小的临时数组，但在 Java 的工程实现中，通常会统一开辟一个大小为 N 的数组供全局使用。所以，它的额外内存开销是 O(N)。
为什么归并排序的空间是 N 而不是 NlogN？
这是最关键的地方。虽然递归树有 logN 层，但计算机并不是同时在执行所有层的合并。
    空间回收： 当计算机执行完“左边一半”的归并后，那一块临时占用的内存就**被释放（或者说可以被覆盖）**了。
    原地复用： 在工业级的代码实现中，我们通常只开辟 一个 和原数组一样大的辅助数组。
        不管你在哪一层合并，你都是把数据从“原数组”搬到这个“辅助数组”，然后再搬回来。
        你永远不需要为每一层都开辟一个新的 N 长度的数组。
    峰值决定复杂度： 空间复杂度看的是内存占用的最高峰。在归并排序的任何一个瞬间，你手里最多就是一个原数组 + 一个辅助数组。
所以，最大空间占用 = N（辅助数组）+ logN（递归产生的栈空间）。由于 N 远大于 logN，所以我们说空间复杂度是 O(N)。

2. 为什么 Random QuickSort 是 O(logN)？
快速排序的重点在于它是一个**原地排序（In-place sort）**算法。
    原地交换：QuickSort 的 partition（分区）操作是直接在原数组上通过 swap（交换）完成的。它不需要像 Mergesort 那样为了合并数据而专门开辟一个 N 大小的临时数组。
    那 O(logN) 是哪来的？：虽然不需要额外的数组，但 QuickSort 是递归调用的。
        每一次递归调用，计算机都需要在内存的**栈（Stack）**里开出一个“空间（Frame）”，用来保存当前处理的数组边界（比如 low 和 high 的下标）。
        在最好或平均情况下（也就是 Random Quicksort），递归树的高度是 logN。
        这意味着在同一时刻，内存栈里最多会有 logN 层递归在“排队”。
    总结：它消耗的内存不是用来存数据的，而是用来记录递归路径的。
