手工制造冲突标准配方

这一招学会了，以后无论是测试 Git 工具，还是给同事演示 Git 原理，都非常有大用。
这里给你总结一份**“手工制造冲突标准配方”**，你可以把它记在心里：
核心原理：制造 "Y" 字形分叉
Git 只有在两条分支同时对同一个文件的同一行做了不同的修改，并且**都已经提交（Commit）**后，才会判定为冲突。
具体步骤总结
1. 制造分身（创建分支）
首先切出一个新分支，作为“另一条时间线”。
Bash
git checkout -b dev  # dev 是随便起的名字
2. 分身搞事情（修改 A）
在 dev 分支上，修改文件 Collatz.java 的第 X 行。 关键点： 改完必须提交！不提交不算数。
Bash
git add .
git commit -m "Change by dev"
3. 回到本体（切回主分支）
回到原来的时间线。
Bash
git checkout main
4. 本体搞事情（修改 B —— 最关键的一步！）
这是你之前卡住的地方。你必须在 main 分支上，也去修改 Collatz.java 的第 X 行（同一个位置），并且改成和刚才不一样的内容。 关键点： 改完也必须提交！
冲突是你俩都改，改完爆炸，如果只是他改
Git 的反应： Git 发现你手里还是第 10 页，而朋友手里是第 10 页 + 第 11 页。 Git 觉得：“这太简单了！你的作业没变，只是他多写了一页。我根本不需要‘合并’，我只要把你手里的作业本换成他那一本就行了。”
这就叫 Fast-forward（快进）。Git 直接把你的进度条“咻”的一下拖到了终点。
Bash
git add .
git commit -m "Change by main"
(如果这一步不做，或者忘了提交，Git 就会直接“快进”合并，不会报错。)
5. 引爆冲突（合并）
现在两边都拿着不同的提交，Git 无法判断谁对谁错。
Bash
git merge dev
结果： CONFLICT (content): Merge conflict... 💥
