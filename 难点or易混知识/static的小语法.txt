1. 外部类（Top-level Class）能带 static 吗？ 
答案：不能。
    为什么： static 的本质含义是“属于类，而不属于某个具体的实例”。
    逻辑矛盾： 外部类已经是代码层级的最高节点了，它不被包含在任何其他类里面。如果你把外部类设为 static，它是相对于谁“静态”呢？这在逻辑上没有意义。
    例外情况： 只有**内部类（Inner Class）**可以被声明为 static。
        非静态内部类： 必须先有外部类的实例，才能创建内部类（它持有外部类的引用）。
        静态内部类（Static Nested Class）： 不需要外部类的实例就能独立存在。这在 BYOW 中很有用，比如你可以在 Room 类里写一个静态的 Point 类。
2. 构造函数（Constructor）能带 static 吗？
答案：绝对不能。
    为什么： 这是由两者的根本目的冲突决定的。
        构造函数的工作： 是为了**创建一个具体的对象（Instance）**并初始化它的状态（给变量赋值）。
        static 的工作： 是为了声明这个东西不需要对象就能访问。
    冲突点： 如果一个构造函数是静态的，你调用它时到底是要创建一个对象，还是不需要对象？这在语义上是完全打架的。构造函数必须与 new 关键字配合使用来产生实例，而 static 刚好是用来“去实例驱动”的。


1. 外部类的 public vs private？
结论：外部类（Top-level Class）只能是 public 或 default（不写修饰符），不能是 private。
    public 类：
        权限： 全世界（任何 package）都能访问它。
        BYOW 应用： 你的 Engine.java 和 Main.java 必须是 public。因为自动评分系统（Autograder）和 Java 运行环境需要从外部包调用它们。
    default（包私有，不写关键词）：
        权限： 只有在同一个 package（文件夹）下的类才能访问它。
        BYOW 应用： 如果你写了一个专门处理“走廊生成逻辑”的辅助类，且不想让其他不相关的包看到它，就可以不写 public。
    为什么不能是 private？ 如果一个外部类是私有的，那么没有任何人能看见它，它也就无法被实例化，那这个类就完全失去了存在的意义。
2. 构造函数的 public vs private？
这才是“高级玩家”经常玩花样的地方。构造函数决定了谁有权 new 出这个对象。
public 构造函数（最常见）
    含义： 任何地方都可以执行 new Room(...)。
    BYOW 应用： 你的 Room 类、Position 类通常都需要 public 构造函数，因为你会在 Engine 类里不断地创建它们。
private 构造函数（进阶用法）
    含义： 外部类无法直接 new 这个对象。只有这个类自己的内部方法可以调用它。
    为什么会有这种“自残”行为？
        工具类（Utility Class）： 比如 Java 自带的 Math 类。你不需要 new Math()，你只需要直接用 Math.abs()。给它一个私有构造函数，可以防止别人不小心实例化它。
        单例模式（Singleton）： 确保整个游戏只有一个“管理器”（比如 GameStateManager）。
        静态工厂方法（Static Factory Method）： 这是我最推荐你在项目里尝试的。
